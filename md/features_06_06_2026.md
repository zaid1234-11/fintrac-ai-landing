### 1.2 Clerk + Supabase Federated Authentication

Financial data is isolated per-user using a two-layer system: **Clerk** for identity management and **Supabase Row-Level Security (RLS)** for database-level enforcement.

**JWT Mapping** — Supabase reads Clerk bearer tokens without a proxy backend via a lightweight SQL function:

```sql
CREATE OR REPLACE FUNCTION public.clerk_user_id()
RETURNS TEXT AS $$
  SELECT nullif(current_setting('request.jwt.claims', true)::json->>'sub', '')::text;
$$ LANGUAGE sql STABLE SECURITY DEFINER;
```

**RLS Policy examples** applied to `users`, `transactions`, `ai_insights`, `behavioral_profiles`, and `historical_budgets`:

```sql
CREATE POLICY "Transactions RLS filter"
ON public.transactions FOR ALL
USING (user_id = public.clerk_user_id());

CREATE POLICY "Users can only read their own profile"
ON public.behavioral_profiles FOR SELECT
USING (user_id = public.clerk_user_id());
```

**User Synchronization** — On `user.created` events, a Svix-verified Clerk Webhook calls `/api/webhooks/clerk`, which inserts the Clerk profile into `public.users` using a Service Role Key (bypassing RLS). All subsequent client calls use only the Anon Key + Clerk JWT.

**Security decisions**:
- No proxy backend — eliminates API latency and reduces attack surface.
- MFA architecturally enabled via Clerk; disabled by default to reduce onboarding friction for early adopters.
- Service Role Key is isolated to the webhook route only.

**Key files**: [`src/middleware.ts`](../src/middleware.ts), `src/utils/supabase/`, `supabase/migrations/00003_security_and_rls.sql`.

### 1.3 Inngest Serverless Event Pipeline

**Registered jobs**:

## 2. Data Ingestion & Preprocessing Engines

### 2.1 Multiline PDF Bank Statement Parser

Extracts individual transactions from uploaded bank statement PDFs. Statements frequently wrap transaction data (date, merchant, amount, running balance) across multiple lines.

**Regex boundary rules**:

- **Transaction start** (date pattern):
  ```
  ^\d+\s+(\d{1,2}\s+[A-Za-z]{3}\s+\d{4}|\d{1,2}[/\-]\d{1,2}[/\-]\d{2,4})\b
  ```
- **Transaction end** (decimal amount + running balance):
  ```
  [\-+]?[\d,]+\.\d{2}\s+[\-+]?[\d,]+\.\d{2}\s*$
  ```

**Debit / Credit direction inference** — When statements omit an explicit direction flag, the parser monitors the running balance delta:

$$\Delta B = B_{\text{current}} - B_{\text{previous}}$$

$$\text{Direction} = \begin{cases} \text{Credit (Income)} & \text{if } \Delta B > 0 \\ \text{Debit (Expense)} & \text{if } \Delta B < 0 \end{cases}$$

**Key files**: `src/lib/parsers/`, `src/lib/jobs/functions.ts` (uploader chunk step).

### 2.2 SMS Normalization & Transaction Extraction

Automatically intercepts and normalizes financial SMS messages from Android clients, extracting clean transaction records from raw notification text.

**Normalization heuristics**:
- **VPA filter** (UPI VPA strings): `[\w.-]+@[\w.-]+` → removed.
- **UTR Reference extraction**: `\b\d{12}\b` → isolated as the transaction reference ID.
- City codes, bank names, and marketing suffixes are stripped before merchant name resolution.

**Key files**: [`src/services/smsSyncService.ts`](../src/services/smsSyncService.ts), [`src/components/SMSSyncStatus.tsx`](../src/components/SMSSyncStatus.tsx).

## 3. 8-Stage Hybrid Transaction Classification Pipeline

Routes each transaction through a cascading set of resolvers, ordered from cheapest (O(1) cache lookup) to most expensive (LLM API call). Each stage short-circuits on a successful match.

```
Raw Transaction
    |
    v
[Clean String Preprocessor]  -- Strips symbols, UPI refs, city tags
    |
    v
[1. User Cache]              merchant_memory table (O(1) exact match)
    |-- Hit --> Done
    |
[2. Global Registry]         merchant_registry table (system-wide)
    |-- Hit --> Done
    |
[3. Heuristic Rules]         Keywords: "Salary", "Rent", "EMI", "Interest"
    |-- Hit --> Done
    |
[4. Fuzzy Levenshtein]       Fuse.js distance matching against known merchants
    |-- Hit --> Done
    |
[5. LLM (GPT-4o-mini)]       Structured JSON output via OpenRouter
    |-- Success --> Done
    |-- Failure -->
[6. Default: "Other"]        Confidence = 0.50
    |
[7. Feedback Loop]           User correction -> updates merchant_memory
```

**Stage details**:

1. **Clean String Preprocessor** — Strips symbols, cleans merchant strings for downstream matching.
2. **User Local Cache (`merchant_memory`)** — Checks user-specific override history first. Zero latency; highest priority.
3. **Global Registry (`merchant_registry`)** — System-wide shared merchant-to-category table.
4. **Heuristic Rule Engine** — Keyword and amount-threshold rules for common fixed categories (Rent, Salary, Loan EMI, Insurance).
5. **Fuzzy Levenshtein Matcher (Fuse.js)** — Computes string similarity between cleaned merchant name and known merchant list. Assigns category when score exceeds threshold.
6. **LLM Fallback (GPT-4o-mini via OpenRouter)** — Structured JSON schema call:
   ```json
   {
     "category": "Dining" | "Shopping" | "Subscriptions" | "Entertainment" | "Other",
     "confidence": 0.0 - 1.0,
     "rationale": "Brief justification text"
   }
   ```
7. **Default Fallback** — Assigns `"Other"` with confidence `0.50` when all stages fail.
8. **Feedback Loop** — User manually corrects a category → API call updates `merchant_memory` and logs an "Active Correction" telemetry event. Next occurrence resolves at Stage 1.

**Key files**: [`src/lib/ai/aiClassifier.ts`](../src/lib/ai/aiClassifier.ts), [`src/lib/ai/classifierPipeline.ts`](../src/lib/ai/classifierPipeline.ts), [`src/lib/ai/pipeline.ts`](../src/lib/ai/pipeline.ts).

## 4. Cryptographic Blockchain Audit Ledger

Guarantees transaction data integrity by chaining every insert into a SHA-256 hash ledger. Prevents retroactive alteration of transaction history (e.g., expense report manipulation, tax log tampering).

**Chaining formula**:

$$\text{Hash}_n = \text{SHA256}(\text{Hash}_{n-1} \parallel \text{ID}_n \parallel \text{Amount}_n \parallel \text{Date}_n)$$

**Genesis block**: `SHA256("genesis_block_" || user_id)` — used when no prior transaction exists.

**SQL trigger** (executes `BEFORE INSERT` on `public.transactions`):

```sql
CREATE OR REPLACE FUNCTION public.process_audit_ledger()
RETURNS TRIGGER AS $$
DECLARE
  prev_hash TEXT;
BEGIN
  SELECT current_hash INTO prev_hash
  FROM public.transactions
  WHERE user_id = NEW.user_id AND id < NEW.id
  ORDER BY id DESC LIMIT 1;

  IF prev_hash IS NULL THEN
    prev_hash := encode(sha256(('genesis_block_' || NEW.user_id)::bytea), 'hex');
  END IF;

  NEW.current_hash := encode(
    sha256((prev_hash || NEW.id::text || NEW.amount::text || NEW.created_at::text)::bytea),
    'hex'
  );

  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

**Key files**: `supabase/migrations/00007_fix_blockchain_trigger.sql`.

## 5. Behavioral Heuristics & Coaching Engine

### 5.1 Time-Decay Weighting

Recent transactions are weighted higher than older history when computing behavioral baselines:

$$w(t) = e^{-0.05 \cdot t}$$

Where $t$ is the transaction's age in days.

### 5.2 Heuristic Alert Triggers

Four alert categories are computed from transaction history and surfaced in the dashboard:

1. **Spending Spike (Anomaly)** — 7-day category spend exceeds ₹2,000 AND is 50%+ above the 3-week weekly average:
   $$\text{Spend}_{7,C} \geq 2000 \quad \text{AND} \quad \frac{\text{Spend}_{7,C} - \text{AvgWeekly}_{21,C}}{\text{AvgWeekly}_{21,C}} > 0.5$$

2. **Category Drift** — Discretionary wallet share increases by ≥15 percentage points in 14 vs. 30 days:
   $$\text{WalletShare}_{14,C} - \text{WalletShare}_{30,C} \geq 0.15$$

3. **Salary Burn Velocity** — Total outflows in the 7 days post-salary credit exceed 50% of the salary amount.

4. **Impulsive Late-Night Spend** — Cumulative debits between 11:00 PM and 5:00 AM exceed ₹1,500.

### 5.3 Financial Wellness Score (4 Pillars)

A global wellness score $W \in [0, 100]$ is computed from four equally-weighted sub-scores:

| Pillar | Formula |
| :--- | :--- |
| **Spending Stability** ($P_{\text{stability}}$) | Base 100, penalized: −10 per Category Drift, −5 per Spending Spike |
| **Savings Consistency** ($P_{\text{savings}}$) | $\max\!\left(0,\; \frac{\text{Credits}_{90} - \text{Debits}_{90}}{\text{Credits}_{90}} \times 100\right)$ |
| **Impulse Pacing** ($P_{\text{impulse}}$) | Base 100, penalized: −2 per Late-Night event, −3 per Weekend Impulse event |
| **Subscription Buffer** ($P_{\text{subscription}}$) | Penalized by fixed recurring outlays relative to total spend |

$$W = \frac{P_{\text{stability}} + P_{\text{savings}} + P_{\text{impulse}} + P_{\text{subscription}}}{4}$$

**Key files**: [`src/lib/ai/behavioralIntel.ts`](../src/lib/ai/behavioralIntel.ts), `src/app/dashboard/page.tsx`.

### 5.4 AI Coaching Engine

Generates supportive, non-judgmental explanations for budget recommendations:
- Explains *why* specific categories were reduced.
- Explains *why* fixed commitments (rent, insurance, loan EMI) are protected.
- Surfaces psychological loss-aversion reframes and a weekly action plan.

**Key file**: [`src/lib/ai/coachingEngine.ts`](../src/lib/ai/coachingEngine.ts).

## 6. Elastic Adaptive Savings Engine

**File**: [`src/lib/ai/elasticSavingsEngine.ts`](../src/lib/ai/elasticSavingsEngine.ts)

Instead of dividing savings cuts equally across categories (which imposes maximum psychological pain on high-friction categories), the Elastic engine allocates cuts inversely proportional to estimated behavioral friction.

### 6.1 Friction-Weighted Cut Allocation

$$\text{Cut}_{t,c} = T_t \cdot \frac{(1 - F_{t,c})^2}{\displaystyle\sum_{c'} (1 - F_{t,c'})^2}$$

Where:
- $T_t$ = total savings target for month $t$
- $F_{t,c} \in [0, 1]$ = estimated behavioral friction for category $c$ at month $t$
- High friction ($F \to 1$) → smaller cut assigned → category is protected
- Low friction ($F \to 0$) → larger cut assigned → category absorbs more

### 6.2 Iterative Water-Filling Distribution

To handle the constraint that a category cannot be cut beyond its actual monthly spend, the engine uses an **iterative water-filling loop** (up to 100 iterations):

1. Identify *active* categories (spending remaining to cut > 0.001).
2. Compute proportional weights from $(1 - F_{t,c})$ for all active categories.
3. Distribute remaining savings target proportionally.
4. For any category where the proposed cut would exceed its remaining spend, cap the cut at the spend limit and carry the surplus back into the next iteration.
5. Repeat until the savings target is met or no active categories remain.

**Edge case**: If all active categories have $F = 1.0$ (weight = 0), cuts are distributed equally among remaining categories.

### 6.3 Behavioral Cost Output

Every optimization call returns a `behavioralCost` metric — the total estimated psychological pain of the proposed cuts:

$$\text{BehavioralCost} = \sum_c \text{Cut}_c \times F_c$$

This is displayed in the Friction Simulator UI as a bar chart, allowing users to see the psychological cost of different savings scenarios before committing.

## 7. Adaptive Friction-Learning Update Engine

**File**: [`src/lib/ai/updateFrictionWeights.ts`](../src/lib/ai/updateFrictionWeights.ts)

This is the core adaptive intelligence of FinTrac AI. It runs monthly (via Inngest `monthlyFrictionUpdate` cron job) and updates each user's per-category friction estimates based on their actual compliance with the previous month's recommended cuts.

> **Terminology note**: This system is *adaptive* and *feedback-driven*, inspired by reinforcement learning principles. It is not classical Q-learning (which requires a discrete state-action table and Bellman updates). The term "adaptive friction-learning" is used throughout.

### 7.1 Compliance Calculation

For each category $c$ in the evaluated month:

$$C_c = \min\!\left(1.0,\; \frac{\text{ActualCut}_c}{\text{SuggestedCut}_c}\right)$$

Where `ActualCut = previousMonthSpend - actualMonthSpend`. A value of `1.0` means the user fully achieved the recommended reduction.

### 7.2 Friction Update Rules

**On non-compliance** ($C_c < 0.99$):

$$F_{c,\,t+1} = \min\!\left(1.0,\; F_{c,t} + \alpha \cdot (1 - C_c) \cdot \text{StreakPenalty}_c\right)$$

Where:
- $\alpha = 0.10$ (learning rate, as implemented in [`updateFrictionWeights.ts`](../src/lib/ai/updateFrictionWeights.ts))
- $\text{StreakPenalty}_c = 1.0 + (\text{consecutive\_failures}_c \times 0.5)$

This means consecutive failures escalate the friction update — the engine learns faster from repeated non-compliance patterns.

**On full compliance** ($C_c \geq 0.99$):
- Failure streak resets to 0.
- Compliance streak increments by 1.
- If `compliance_streak >= 3` consecutive months: $F_{c,t+1} = \max(0.0,\; F_{c,t} - 0.05)$
- Otherwise: friction is held constant (reinforced without decay).

**Cold start default**: New or unprofiled categories initialize at $F = 0.5$ (neutral uninformed prior).

## 9. Academic Simulation & Validation Framework

### 9.1 Monte Carlo POMDP Simulation

**Purpose**: Validate the theoretical advantage of the Elastic Adaptive Savings Engine against Traditional proportional budgeting, under controlled but realistic behavioral conditions.

**Experimental parameters** (baseline run):

| Parameter | Value |
| :--- | :--- |
| Cohort size ($N$) | 5,000 synthetic agents |
| Simulation horizon ($T$) | 12 months |
| Deterministic seed | 42 |
| Learning rate ($\alpha$) | 0.15 (simulation), 0.10 (production engine) |
| Churn threshold | Compliance < 30% for 3 consecutive months |

**Behavioral personas** (manually specified design assumptions — not data-discovered):

| Persona | Share | Behavioral Model |
| :--- | :---: | :--- |
| Aspirational Saver | 40% | Moderate initial friction; compliance improves +0.02/month (habit formation) |
| Income Shock | 30% | At Month 6: friction increases +0.10 across all categories, target reduced to 75% |
| Seasonal Spender | 30% | Month 11: Shopping friction surges to 0.95; Month 12: resets to 0.50 |

**POMDP noise model**:
- Observation noise: $\mathcal{N}(0, 0.05)$ added to compliance signals seen by the engine.
- Execution noise: $\mathcal{N}(0, 0.10)$ added to actual spending, modeling everyday financial variation.

### 9.2 Simulation Results (N=5,000, T=12 months, seed=42)

> ⚠️ These are **simulation outcomes**, not real-world measurements.

| Metric | Adaptive RL (Simulated) | Traditional (Simulated) | Statistical Test |
| :--- | :---: | :---: | :--- |
| Month 12 Retention | **97.08%** | 63.54% | $\chi^2(1) = 1723.1,\; p < 0.001$ |
| Mean Active Lifespan | **11.78 months** | 9.56 months | $t(9821) = 52.12,\; p < 0.001$ |

### 9.3 Ablation Study Results

Three-way comparison isolating the contribution of active adaptation vs. initial profiling quality:

| System | Initial Prior | Active Adaptation | Simulated Month 12 Retention |
| :--- | :---: | :---: | :---: |
| Traditional Budgeting | Equal weights | No | 62.82% |
| Static Friction-Aware | Perfect true baseline | No | 68.58% |
| **Adaptive Engine** | Uninformed (0.5) | **Yes** | **97.90%** |

**Key finding**: Dynamic adaptation outperforms even perfect static profiling under modeled shock conditions. A static model with perfect initial knowledge fails when users experience Income Shocks or Seasonal Spender surges — because it cannot reroute cuts after the environment changes.
