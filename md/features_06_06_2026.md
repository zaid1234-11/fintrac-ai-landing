### 1.2 Clerk + Supabase Federated Authentication
- **Description**: Financial data is securely isolated using Row-Level Security (RLS) policies in Supabase, linked to Clerk's identity templates.
- **JWT Mapping**: Custom SQL helper `public.clerk_user_id()` parses the decoded `sub` claim of Clerk’s bearer tokens:
  ```sql
  CREATE OR REPLACE FUNCTION public.clerk_user_id()
  RETURNS TEXT AS $$
    SELECT nullif(current_setting('request.jwt.claims', true)::json->>'sub', '')::text;
  $$ LANGUAGE sql STABLE SECURITY DEFINER;
  ```
- **RLS Policies**: Applied to tables `users`, `transactions`, `ai_insights`, `behavioral_profiles`, and `historical_budgets` to restrict reads/writes to `user_id = public.clerk_user_id()`.
- **User Synchronization**: Implemented via a Clerk Webhook route handler verifying Svix signatures to insert Clerk users into Supabase's `public.users` table on registration.

## 2. Ingestion & Preprocessing Engines

### 2.1 Multiline PDF Bank Statement Parsing
- **Description**: Extracts transaction details from uploaded bank statements. It uses regex parsing to scan for multi-line transactions wrapped across pages.
- **Regex Boundary Rules**:
  - Starts parsing when it matches a date: `^\d+\s+(\d{1,2}\s+[A-Za-z]{3}\s+\d{4}|\d{1,2}[/\-]\d{1,2}[/\-]\d{2,4})\b`
  - Finishes transaction block when it matches decimal amounts and running balances: `[\-+]?[\d,]+\.\d{2}\s+[\-+]?[\d,]+\.\d{2}\s*$`
- **Resolution**: Computes debit/credit direction based on difference in running balances:
  $$\Delta B = B_{\text{current}} - B_{\text{previous}}$$
  If $\Delta B > 0$, transaction is `credit`; if $\Delta B < 0$, it is `debit`.
- **Files**: `src/lib/parsers/`, `src/lib/jobs/functions.ts` (uploader chunk step).

### 2.2 SMS Normalization & Transaction Extraction
- **Description**: Automatically intercepts, normalizes, and logs transactions from financial SMS messages received on Android clients.
- **Heuristics**: Strips transaction references, UPI VPAs, and city tags from description fields:
  - VPAs cleaned: `[\w.]+@[\w.]+` $\rightarrow$ removed.
  - Ref codes: isolates 12-digit UTR identifiers (`\b\d{12}\b`).
- **Files**: [smsSyncService.ts](file:///C:/Users/zaids/.gemini/antigravity/scratch/fintrac-ai-landing/src/services/smsSyncService.ts), [SMSSyncStatus.tsx](file:///C:/Users/zaids/.gemini/antigravity/scratch/fintrac-ai-landing/src/components/SMSSyncStatus.tsx).

## 3. 8-Stage Hybrid Transaction Classification Pipeline

Balances speed and cognitive reasoning by routing transactions through 8 distinct filters:

```
Raw Transaction ──> Clean String ──> 1. Local Cache Lookup (merchant_memory)
                                           ├── [Hit] ──> Done
                                           └── [Miss] ─> 2. Global Registry Lookup
                                                              ├── [Hit] ──> Done
                                                              └── [Miss] ─> 3. Heuristic Rules (e.g. Rent, Salary)
                                                                                  ├── [Hit] ──> Done
                                                                                  └── [Miss] ─> 4. Fuzzy Levenshtein (Fuse.js)
                                                                                                      ├── [Hit] ──> Done
                                                                                                      └── [Miss] ─> 5. LLM (GPT-4o-Mini)
                                                                                                                          ├── [Success] ─> Done
                                                                                                                          └── [Fallback] ─> 6. Default ("Other")
```

1. **Clean String Preprocessor**: Strips symbols and metadata from transaction titles.
2. **User Local Cache (`merchant_memory`)**: Scans user-specific override histories ($O(1)$).
3. **Global Registry (`merchant_registry`)**: System-wide lookup matching shared merchant categories.
4. **Heuristic Rule Engine**: Applies keyword matching and amount thresholds:
5. **Fuzzy String Matcher**: Runs a Levenshtein distance check via Fuse.js against known merchants.
6. **LLM Fallback (OpenRouter GPT-4o-Mini)**: Calls LLM with strict JSON formatting to resolve category and confidence scores.
7. **Default Fallback**: Categorizes as `Other` with a fallback confidence of `0.50`.
8. **Feedback Loop**: When users manually modify a transaction category in the UI, an API call updates the local memory cache and registers an "Active Correction" in telemetry.

## 4. Cryptographic Blockchain Audit Ledger

- **Description**: Guarantees transaction data integrity by logging every entry in a cryptographic chain. Prevents retroactively altering history (e.g., for expense report manipulation or tax logs).
- **Chaining Heuristic**: Applies SQL triggers (`BEFORE INSERT` and `AFTER INSERT`) where the current transaction's hash is computed as:
  $$\text{Hash}_n = \text{SHA256}(\text{Hash}_{n-1} \parallel \text{ID}_n \parallel \text{Amount}_n \parallel \text{Date}_n)$$
- **Genesis Block**: Starts with a genesis string: `SHA256("genesis_block_" || user_id)`.
- **Files**: Migration file `00007_fix_blockchain_trigger.sql`, database triggers in Supabase.

## 5. Behavioral Heuristics & Coaching Engine

### 5.1 Time-Decay Weighting Function
To weight recent transactions higher than older history when analyzing habits, Fintrac AI applies a time-decay factor:
$$w(t) = e^{-0.05 \cdot t}$$
Where $t$ is the age of the transaction in days.

### 5.2 Heuristic Alert Triggers
1. **Spending Spikes (Anomalies)**: 7-day spending in category $C$ exceeds ₹2,000 and is 50%+ higher than the 3-week weekly average:
   $$\text{Spend}_{7, C} \ge 2000 \quad \text{AND} \quad \frac{\text{Spend}_{7, C} - \text{AvgWeekly}_{21, C}}{\text{AvgWeekly}_{21, C}} > 0.5$$
2. **Category Drift**: Discretionary wallet share shifts upwards by $\ge 15\%$ points:
   $$\text{WalletShare}_{14, C} - \text{WalletShare}_{30, C} \ge 0.15$$
3. **Salary Burn Velocity**: Total outflows in the 7 days post-salary exceed $50\%$ of the salary credit.
4. **Impulsive Late-Night Spend**: Cumulative debits between 11:00 PM and 5:00 AM exceed ₹1,500.
- **Files**: `src/lib/ai/behavioralIntel.ts`.

### 5.3 Financial Wellness Score Pillars
Financial Wellness is modeled across four distinct pillars ($P \in [0, 100]$) aggregated into a global score:
- **Spending Stability ($P_{\text{stability}}$)**: Penalized by Category Drifts (-10 each) and Spikes (-5 each).
- **Savings Consistency ($P_{\text{savings}}$)**: Net savings ratio:
  $$P_{\text{savings}} = \max\left(0, \frac{\text{Credits}_{90} - \text{Debits}_{90}}{\text{Credits}_{90}} \cdot 100\right)$$
- **Impulse Pacing ($P_{\text{impulse}}$)**: Penalized by Late-Night (-2 each) and Weekend Impulse spending (-3 each).
- **Subscription Buffer ($P_{\text{subscription}}$)**: Penalized by fixed recurring outlays relative to total spending.
- **wellness_score**: $\frac{P_{\text{stability}} + P_{\text{savings}} + P_{\text{impulse}} + P_{\text{subscription}}}{4}$
- **Files**: `src/lib/ai/behavioralIntel.ts`, `src/app/dashboard/page.tsx` (Wellness Audit UI).

## 6. Reinforcement Learning Savings Optimization Engine (Elastic RL)

Instead of cutting evenly across categories, the engine allocates reductions based on behavioral friction and adapts to compliance in real-time.

### 6.1 Proportional Elastic Cuts (Water-Filling Optimization)
- **Friction Score ($FS_i$)**: Composite score of frequency, Consistency (StdDev Coefficient of Variation), and positive sentiment:
  $$FS_i = 0.4 \cdot \text{Frequency} + 0.3 \cdot (1 - \text{CV}) + 0.3 \cdot \text{Sentiment}$$
- **Reduction Weight**: Distributed inversely proportional to friction:
  $$\text{ReductionWeight}_i = \frac{1 - FS_i}{\sum_j (1 - FS_j)}$$
- **Iterative Capping**: If a suggested cut exceeds a category's monthly spend, the cut is capped at the spending limit, and the excess savings target is redistributed among the remaining active categories.
- **Files**: [elasticSavingsEngine.ts](file:///C:/Users/zaids/.gemini/antigravity/scratch/fintrac-ai-landing/src/lib/ai/elasticSavingsEngine.ts).

### 6.2 Reinforcement Learning Friction Update Rule
- **Compliance ($C_i$)**: Evaluates how closely a user followed the recommended cut:
  $$C_i = \min\left(1.0, \frac{A_i}{P_i}\right)$$
- **Friction Update (Q-Learning Style)**: Runs monthly to update weights:
  $$F_{i, t+1} = \min\left(1.0, F_{i, t} + \alpha \cdot (1 - C_i) \cdot \text{StreakPenalty}_i\right)$$
  - Learning rate $\alpha = 0.10$.
  - Streak penalty $\text{StreakPenalty}_i = 1.0 + (\text{consecutive\_failures}_i \cdot 0.5)$.
- **Habit Formation Recovery Decay**: If user maintains perfect compliance ($C_i \ge 0.99$) for $\ge 3$ consecutive months, friction decays by `-0.05` to reward habit change.
- **Cold Start**: Unprofiled categories default to a neutral prior of `0.5`.
- **Files**: [updateFrictionWeights.ts](file:///C:/Users/zaids/.gemini/antigravity/scratch/fintrac-ai-landing/src/lib/ai/updateFrictionWeights.ts).

## 8. Academic Simulation & Validation Framework

### 8.1 Noisy POMDP & Personas Model
To validate the system's efficacy empirically, the project incorporates a Monte Carlo simulation framework.
- **Personas**: Aspirational Saver, Income Shock (retains 75% capacity at Month 6 with a +0.10 friction spike), and Seasonal Spender (Month 11 shopping rigidity surge).
- **Signal Noise (POMDP)**: Adds standard Gaussian noise ($\mathcal{N}(0, 0.05)$) to the compliance signals observed by the agent, modeling incomplete visibility of transactional compliance.
- **User Churn Logic**: Users who maintain compliance $< 30\%$ for three consecutive months are permanently deactivated (churned).
- **Personalized Baseline & Relative Recovery**: Computes pre-shock compliance averages (Months 1–5). A shocked user achieves recovery if compliance matches $\ge 90\%$ of this baseline for 2 consecutive months.
- **Files**: [fintrac_simulation_final.py](file:///C:/Users/zaids/.gemini/antigravity/scratch/fintrac-ai-landing/fintrac_simulation_final.py), [extract_results.py](file:///C:/Users/zaids/.gemini/antigravity/scratch/fintrac-ai-landing/extract_results.py).

### 8.2 Simulation Verification Results (Mild Shock POMDP)
- **Cohort Results (N=5,000, T=12 months)**:
  - **Traditional Budgeting (Control)**: Month 12 retention of **63.54%** with a mean active lifespan of **9.56 months**.
  - **Elastic RL (Treatment)**: Month 12 retention of **97.08%** with a mean active lifespan of **11.78 months**.
  - **Compliance Ratio**: Elastic RL achieved **41.63%** compliance vs. **38.61%** for Traditional ($t(9821) = 55.86, p < 0.001$, Cohen's $d = 0.34$).
  - **Resilience / Relative Recovery**: Under a mild income shock (25% capacity reduction at Month 6), the RL engine successfully recovered **14.43%** of users back to $\ge 90\%$ of their pre-shock baseline within an average of **3.66 months** post-shock.

### 8.3 Ablation Study Results (Deterministic Seed 42)
To isolate the contribution of dynamic reinforcement learning relative to static profiling, a three-way ablation experiment was executed (incorporating corrected compliance evaluations, a 45% aggregate churn threshold, and a seasonal spender trap):
1. **Traditional Budgeting**: No initial friction profiling (equal weights) and no active learning.
   - **Month 12 Retention**: **52.88%**
2. **Static Friction-Aware**: Perfect initial friction profiling (starts with true user baseline) but no learning (weights cannot adapt to shocks or habit changes).
   - **Month 12 Retention**: **71.14%**
3. **Adaptive RL Engine**: No initial profiling (uninformed prior of `0.5`) but active reinforcement learning (adapts to shocks and habits).
   - **Month 12 Retention**: **96.42%**
- **Discussion**: The ablation results demonstrate that **dynamic adaptation (RL) is significantly more important than perfect initial profiling (Static)**. Even though the Static model starts with perfect knowledge of the user's true baseline, its inability to adapt to shocks (e.g. Income Shock at Month 6) or habit rigidity (e.g. Seasonal Spender at Month 11) causes its performance to decay over time, leading to a 71.14% retention rate. The Adaptive RL engine, starting with an uninformed prior (0.5), successfully learns the environment dynamics and adapts to behavioral shifts, achieving a **96.42%** retention rate.
